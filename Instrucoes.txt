--------------------------------------------------------------------------------
Para lidar com uma lista extensa de equipamentos (importados do Excel), foi
implementada uma busca por digitação (autocomplete) local/em memória no campo de seleção de máquinas.

A. Modificações no Backend:
--------------------------
1. O formulário Django `StartServiceForm` em `forms.py` carrega a lista completa de
   máquinas através do queryset `Machine.objects.select_related('setor').order_by('nome')`
   para renderizar todas as opções diretamente nas tags `<option>` do HTML.
2. A utilização de `select_related('setor')` otimiza o carregamento evitando o problema de
   consultas adicionais (N+1) no banco de dados.
3. O texto exibido para o usuário mostra a combinação máquina + setor no formato:
   `"Nome da Máquina [Setor: Nome do Setor]"`.

B. Modificações no Frontend (Templates):
---------------------------------------
1. No layout base `base.html`, o bloco `extra_css` foi realocado para fora da
   tag `<style>`, possibilitando a inclusão de tags `<link>` de folha de estilo CDN.
2. Na tela de "Controle de Técnicos" (`technician_management.html`):
   - Foram integrados os links CDN oficiais (CSS e JS) da biblioteca **Tom Select**.
   - O Tom Select é inicializado em modo estático local no campo `#id_maquina`:
     * O dropdownParent está definido como `'body'` para evitar que o dropdown seja cortado
       pelas bordas/overflow do modal do Bootstrap.
     * **Exibição Imediata:** Ao clicar no campo, todas as opções carregadas aparecem imediatamente,
       permitindo ao operador rolar a lista ou digitar para filtrar em tempo real no próprio frontend.
     * Ao reabrir o modal "Iniciar Serviço" para qualquer técnico, a seleção anterior e a busca
       são limpas automaticamente.

C. Correção de Sintaxe do CSS Exposto:
-------------------------------------
1. Como o bloco `extra_css` em `base.html` foi movido para fora da tag `<style>` global, os templates
   `login.html` e `tv_dashboard.html` que sobrescreviam este bloco começaram a renderizar suas regras de CSS
   bruto como texto puro no topo da página.
2. Este problema foi corrigido com sucesso encapsulando explicitamente as regras de CSS destes templates
   dentro de tags `<style></style>` válidas no bloco `extra_css`.
3. Com isso, todo o vazamento de texto foi removido, o layout do Painel voltou ao normal e a inicialização
   do JavaScript do buscador (Tom Select) ocorre perfeitamente sem erros de travamento.


9. SUPORTE A MÚLTIPLAS ALOCAÇÕES E SERVIÇOS PAUSADOS
--------------------------------------------------------------------------------
Esta seção descreve a lógica de concorrência de alocações implementada para permitir
que um técnico acumule mais de um serviço em aberto ao mesmo tempo.

A. Regras de Negócio:
---------------------
- Um técnico pode ter NO MÁXIMO 1 alocação com status 'EM_ATENDIMENTO' por vez.
- Um técnico pode ter ILIMITADAS alocações com status 'EM_PAUSA' simultaneamente.
- O status do model Technician (.status) sempre reflete o estado do serviço ativo:
  * 'EM_ATENDIMENTO' → há exatamente 1 alocação aberta com status='EM_ATENDIMENTO'
  * 'EM_PAUSA'       → não há ativo, mas há ao menos 1 pausada
  * 'OCIOSO'         → não há nenhuma alocação aberta (todas encerradas com data_fim)

B. Novo Campo no Model Allocation (models.py):
----------------------------------------------
- Adicionado o campo `status` (CharField) no model Allocation com as seguintes opções:
  * 'EM_ATENDIMENTO' (padrão ao criar)
  * 'EM_PAUSA'
- Migração gerada: `maintenance/migrations/0002_allocation_add_status.py`
- Comando para reaplicar se necessário: python manage.py migrate

C. Propriedades Corrigidas/Adicionadas em Technician (models.py):
-----------------------------------------------------------------
- `active_allocation`: filtra allocations onde data_fim IS NULL AND status='EM_ATENDIMENTO'
  (antes filtrava apenas data_fim IS NULL — podia retornar pausadas erroneamente)
- `paused_allocations`: nova propriedade que retorna QuerySet de alocações onde
  data_fim IS NULL AND status='EM_PAUSA', ordenadas por data_pausa

D. Lógica das Views (views.py):
--------------------------------
- `start_service`: NÃO bloqueia se o técnico tiver apenas serviços pausados. Bloqueia
  SOMENTE se já houver uma alocação EM_ATENDIMENTO ativa (technician.active_allocation is not None).
- `pause_service`: seta allocation.status='EM_PAUSA' na alocação ativa e recalcula
  o status do técnico (vai para 'EM_PAUSA' se não houver outra ativa restante).
- `resume_service`: retoma a alocação pausada mais antiga — caso simples (sem ativa).
- `finish_service`: finaliza a alocação EM_ATENDIMENTO e recalcula status do técnico
  (EM_PAUSA se restarem pausadas, OCIOSO se não restar nenhuma).
- `resume_paused_allocation(allocation_id)` [NOVA]: troca de contexto — ativa a alocação
  pausada selecionada; se havia uma EM_ATENDIMENTO, ela vai automaticamente para EM_PAUSA
  com motivo "Interrompido para retomada de outro serviço."
- `finish_allocation(allocation_id)` [NOVA]: finaliza uma alocação ESPECÍFICA (ativa ou pausada)
  pelo seu ID. Recalcula o status do técnico com base nas alocações restantes.

E. Novas Rotas de URL (urls.py):
---------------------------------
  allocations/<int:allocation_id>/resume/  → resume_paused_allocation (name='resume_paused_allocation')
  allocations/<int:allocation_id>/finish/  → finish_allocation         (name='finish_allocation')

F. Interface do Operador (technician_management.html):
------------------------------------------------------
- Card do técnico redesenhado:
  * Seção "Serviço Ativo" exibe a alocação EM_ATENDIMENTO.
  * Badge "+N pausado(s)" aparece no header se houver alocações pausadas.
  * Seção colapsável "Serviços Pausados (N)" lista cada alocação pausada com:
    - Nome da máquina, criticidade, tempo decorrido, atividade, motivo da pausa.
    - Botão "Retomar" → POST para resume_paused_allocation/<allocation_id>/ (troca de contexto).
    - Botão "Finalizar" → abre modal #finishAllocModal com action apontando para
      finish_allocation/<allocation_id>/.
  * Botão "Iniciar Serviço" visível mesmo quando o técnico tem serviços pausados (mas não ativo).
- Modal #finishAllocModal: equivalente ao #finishModal, mas dinâmico por allocation_id.
  O campo action do form é setado via JavaScript ao abrir o modal.

11. STATUS DE AUSÊNCIA E DISPONIBILIDADE DO TÉCNICO
--------------------------------------------------------------------------------
Esta seção documenta a expansão dos status do modelo Technician para contemplar
estados de ausência e situações de escala fora da fábrica.

A. Novos Status Adicionados (models.py — Technician.STATUS_CHOICES):
---------------------------------------------------------------------
Código              | Rótulo                              | Situação
--------------------|-------------------------------------|------------------------------
OCIOSO              | Disponível (Ocioso)                 | Técnico na fábrica, disponível
EM_ATENDIMENTO      | Em Atendimento                      | Serviço ativo em campo
EM_PAUSA            | Em Pausa                            | Serviço interrompido
AUSENTE_FOLGA       | Ausente – Folga/Escala              | Folga ou escala no dia
AUSENTE_FERIAS      | Ausente – Férias                    | Em período de férias
AUSENTE_MEDICO      | Ausente – Licença Médica/Afastamento| Afastado por saúde
EXTERNO_PLANTAO     | Plantão Fora da Fábrica             | Plantão em outra unidade

B. Constante de Controle (Technician.STATUS_AUSENCIA):
------------------------------------------------------
O model define o atributo de classe:
    STATUS_AUSENCIA = {'AUSENTE_FOLGA', 'AUSENTE_FERIAS', 'AUSENTE_MEDICO', 'EXTERNO_PLANTAO'}

E a propriedade:
    @property is_ausente → bool   (True se status ∈ STATUS_AUSENCIA)

C. Regras de Negócio:
--------------------
- Técnico com status de ausência NÃO pode receber novas ordens de serviço.
  A view start_service bloqueia a operação e exibe mensagem de erro.
- Serviços em EM_PAUSA existentes são MANTIDOS congelados no histórico; os
  botões "Retomar" ficam desabilitados enquanto o técnico estiver ausente.
- O status EM_ATENDIMENTO não pode ser definido manualmente via widget de
  disponibilidade — ele é controlado exclusivamente pelas views start/pause/finish.
- Um técnico EM_ATENDIMENTO não pode ser marcado como ausente diretamente;
  o operador deve primeiro pausar ou finalizar o serviço ativo.

D. Nova View e Rota (views.py / urls.py):
-----------------------------------------
- View: set_availability(request, technician_id)
- URL : POST /technicians/<id>/availability/   (name='set_availability')
- Aceita: OCIOSO, AUSENTE_FOLGA, AUSENTE_FERIAS, AUSENTE_MEDICO, EXTERNO_PLANTAO
- Para retornar o técnico ao fluxo normal, selecionar "Disponível (Ocioso)".

E. Interface do Operador (technician_management.html):
------------------------------------------------------

13. HISTÓRICO MULTIPLO DE PAUSAS E RETORNOS (RELACIONAL)
--------------------------------------------------------------------------------
Para evitar a perda de dados ao retomar um atendimento e permitir múltiplos registros
de pausa em uma única alocação, foi criada uma tabela relacional de histórico.

A. Novo Modelo de Banco de Dados (HistoricoPausa):
--------------------------------------------------
Campos definidos no model HistoricoPausa:
- alocacao     -> ForeignKey para o model Allocation (related_name='pausas')
- data_pausa   -> DateTimeField (registra o momento de início da pausa)
- data_retorno -> DateTimeField (nulo ao pausar, preenchido com timezone.now() ao retomar)
- motivo_pausa -> TextField (obrigatório, contendo o motivo da pausa)

B. Comportamento nas Views (views.py):
-------------------------------------
- Pausar: Cria um registro no HistoricoPausa com o timestamp atual e o motivo. Altera o status da alocação para 'EM_PAUSA'.
- Retomar: Localiza o último registro no HistoricoPausa onde 'data_retorno' é nulo e preenche com a data/hora atual. Altera o status para 'EM_ATENDIMENTO'.
- Concluir: Além de definir 'data_fim' no registro principal de Allocation, se houver qualquer pausa em aberto, a data_retorno da pausa é preenchida automaticamente com o mesmo horário de fim de serviço, fechando o ciclo de forma consistente.

C. Exibição no Django Admin (admin.py):
---------------------------------------
- Foi implementado um TabularInline (HistoricoPausaInline) na classe AlocacaoAdmin.
- Ao visualizar ou editar uma Alocação antiga, uma tabela completa e elegante do histórico de pausas do técnico é renderizada diretamente na página de detalhes.
